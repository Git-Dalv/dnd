# VTT — виртуальный игровой стол для D&D

Десктопное приложение для игры в D&D с компанией друзей: карта с токенами,
броски кубиков, карточки персонажей, видеосвязь, музыка/эффекты от мастера.
Цель — «всё в одном» и удобство ведения, гибрид классического VTT и UI в духе BG3.
Не продукт на рынок, а инструмент для себя и друзей (4–6 человек).

## Стек

- **Backend:** Python 3.12, FastAPI, WebSocket (нативный, через `websockets`/Starlette). Состояние в памяти на старте, БД позже.
- **Frontend:** начинаем с одного HTML/JS (прототип уже есть в `docs/`), позже — фреймворк по вкусу. Десктоп — потенциально Tauri.
- **Видео:** WebRTC mesh (P2P) на 4–6 человек, сервер только для signaling. NAT решается через Tailscale (вся компания в одной приватной сети — TURN не нужен).
- **Подключение:** сервер у хоста, друзья заходят по адресу в Tailscale-сети.

## Главный архитектурный принцип

**Сервер — единственный источник истины. Клиент шлёт намерения, сервер рассылает подтверждённые изменения.**
Клиент никогда не меняет состояние «на веру» и никогда не генерит результаты сам.

Из этого следует:
- **Броски кубиков считает СЕРВЕР.** Клиент шлёт `dice.roll` (без результата), сервер генерит число и рассылает `dice.rolled`. Клиент только проигрывает анимацию (BG3-кубики) по готовому числу. Иначе игрок подделает d20.
- **Скрытое — фильтруется на сервере ДО отправки.** Скрытые токены, нераскрытый туман, приватные разговоры не отправляются тем, кому не положено. Не «отправить и не показать» — иначе видно через devtools.
- **Проверка прав на сервере.** GM-only действия (смена сцены, туман, старт боя, музыка) сервер игнорирует от игроков. `token.move` принимается только от владельца токена или GM.

## Протокол событий

Два потока. Имена намерений — глаголы (клиент→сервер), трансляций — факт в прошедшем (сервер→все).

Клиент → сервер (намерения):
- `dice.roll` { notation, modifier, label }
- `token.move` { tokenId, to:{x,y} }
- `action.use` { charId, actionId }
- `scene.switch` { sceneId }            // gm only
- `fog.reveal` { cells:[[x,y]] }         // gm only
- `combat.start` { participants }        // gm only
- `combat.next` {}                       // gm only
- `audio.play` { channel, track }        // gm only
- `request.toGm` { kind, label }         // игрок → приватно мастеру
- `whisper.open` { audience:[playerId], label }  // gm only, открывает приватный канал

Сервер → все (трансляции):
- `dice.rolled` { by, label, rolled:[..], total }
- `token.moved` { tokenId, to, by }
- `scene.switched` { sceneId, state }
- `audio.state` { channel, track, playing }
- `combat.updated` { combat }
- `member.connection` { memberId, connected }

Приватные (сервер → только audience): `whisper.*`, `request.toGm` (→ только gm).

## Координаты

Токены хранятся в КЛЕТКАХ (x:5, y:8), не в пикселях. Пиксели считает клиент из grid.size при отрисовке. Привязка к сетке — бесплатно.

## Карточка vs Персонаж

Одни данные, два представления:
- **Карточка** (левая вкладка) — полный лист, все характеристики, привычный вид.
- **Персонаж** (таб в hotbar) — компактная выжимка для боя.
Меняется одно поле — меняется везде. НЕ держать две копии данных.

## Универсальный объект roll

У навыка, атаки, урона — одна форма: { notation, modifier, type }.
Панель и движок кубиков не знают, откуда бросок. Это даёт совместимость:
ручной ввод сейчас и импорт из SRD позже ложатся в одну структуру.
`modifier` хранится посчитанным (не на лету).

## Данные D&D 5e — правовое

Механику 5e (числа, тексты заклинаний) можно брать из SRD под OGL:
репозиторий `5e-bits/5e-database` (JSON). Контент из платных книг (PHB и т.п.) — нельзя.
Для «себя и друзей» не критично, но наполнять — из SRD, не копипастить из книг.

## Порядок разработки (MVP-first)

1. **Вертикальный срез:** кубик от клика → сервер → журнал, между двумя вкладками браузера. Даёт всю шину событий.
2. Карточки персонажей (JSON, ручной ввод/импорт), общий лог.
3. Карта с токенами, синхронизация движения, fog of war.
4. Инициатива/бой.
5. Музыка/эффекты от мастера.
6. Видео (WebRTC mesh) внутрь.
7. Запросы к DM с выбором аудитории (приватные каналы).
8. Позже: редактор карт, импорт из SRD, автозаполнение способностей.

Каждую фичу делать минимально-достаточной и сразу играть на ней.
