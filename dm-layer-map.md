# DM-слой — архитектурная карта и поэтапный план

Документ описывает ВЕСЬ слой мастера целиком (чтобы блоки не разъехались), затем
режет на этапы для поэтапной реализации. Каждый этап = отдельная серия промптов
под Claude Code; берём по одному.

## Фундаментальный принцип слоя

DM — не «игрок с лишними кнопками», а ОТДЕЛЬНАЯ РОЛЬ с другими правами. В CLAUDE.md
уже есть is_gm и проверка прав на сервере. Отсюда два сквозных правила:

1. ОДИН prototype.html рендерит разный UI по роли (role: gm|player из лобби).
   НЕ делать отдельную страницу DM — разъедется. Клиент ветвит рендер; игроку —
   бар слотов, мастеру — «пульт мастера».
2. Клиент может соврать — СЕРВЕР источник истины. Любое DM-действие (инициировать
   проверку, двигать NPC, сменить сцену) сервер принимает ТОЛЬКО если отправитель
   is_gm. Рисование DM-кнопок у игрока ничего не даёт: сервер отклонит.

DM НЕ имеет бара способностей-заклинаний — он не играет персонажем, он управляет
сценой. Его нижняя зона — «пульт мастера» вместо слотов.

═══════════════════════════════════════════════════════════════════════
БЛОК 1 — Инициирование проверок и спасбросков (модель A+)
═══════════════════════════════════════════════════════════════════════

Центральная DM-механика. Модель A с дополнениями: DM ПРОСИТ, игрок КИДАЕТ сам.

Поток:
- DM на пульте выбирает: кого (один/несколько/вся партия), что (спасбросок
  характеристики / проверка навыка / инициатива), сложность DC (число или «без DC»).
- Сервер создаёт ЗАПРОС ПРОВЕРКИ с жизненным циклом (pending) для каждого адресата
  и шлёт его адресным игрокам. Запрос — это СОСТОЯНИЕ, не разовое сообщение:
  сервер хранит висящие запросы, чтобы свёрнутое окно пережило перезагрузку
  страницы игрока (при reconnect игрок получает свои pending-запросы снова).
- У игрока открывается окно: «Мастер просит: Спасбросок Ловкости, DC 15».
  Окно МОЖНО СВЕРНУТЬ, но НЕЛЬЗЯ ЗАКРЫТЬ — запрос нельзя проигнорировать, висит
  свёрнутым напоминанием, пока не брошен.
- Игрок жмёт «Бросить» — клиент шлёт намерение, СЕРВЕР кидает (как dice.roll,
  сервер источник броска), прибавляет нужный модификатор персонажа (спасбросок/
  навык берётся из карточки), сравнивает с DC → успех/провал.
- ПЕРЕБРОС за вдохновение: если у персонажа heroicInspiration=true, в окне есть
  «Перебросить (вдохновение)». Сервер списывает вдохновение из resources и
  перекидывает d20, заменяя результат (вдохновение — ресурс, источник истины сервер).
- Результат уходит DM (таблица «Лиэндра 18 ✓, Торин 9 ✗») и в журнал.

Серверное состояние: pending-запросы по комнате { reqId, to:[memberIds], kind,
ability/skill, dc, results:{memberId: {roll, total, success}} }. События:
  check.request (DM→сервер→адресатам), check.roll (игрок→сервер),
  check.result (сервер→DM + журнал), check.reroll (игрок, тратит вдохновение).
Права: check.request принимает только is_gm.

═══════════════════════════════════════════════════════════════════════
БЛОК 2 — NPC / фишки неигровых персонажей
═══════════════════════════════════════════════════════════════════════

DM добавляет на карту фишки NPC. Фишка = ПОЛНЫЙ персонаж (подмножество
character-schema.json), управляемый DM.

- На карте фишка показывает только ПОРТРЕТ/иконку. По клику открывается мини-лист:
  HP, AC, тип существа, характеристики, действия/атаки. Переиспользует рендер
  карточки персонажа (та же схема — НЕ выдумывать отдельную структуру, иначе бой
  придётся переделывать).
- Статблок NPC хранится как персонаж с пометкой npc:true, владелец — DM.
- РАЗДЕЛЕНИЕ ПРАВ ТОКЕНОВ: у токена появляется controlledBy (memberId/'gm').
  token.move сервер принимает только от владельца: токен игрока двигает игрок,
  токен NPC — только DM. Это расширяет пункт 3 (токены) разграничением прав.
- Связь с боем (пункт 4): NPC со статблоком станут участниками трекера инициативы.
  Поэтому статблок сразу в форме character-schema (hp/ac/initiative/actions).

Серверное: токены получают поле controlledBy; npc-персонажи хранятся как
characters с npc=true, owner=gm. События: npc.create (DM), npc.update,
token.move (с проверкой controlledBy).

═══════════════════════════════════════════════════════════════════════
БЛОК 3 — Управление сценой
═══════════════════════════════════════════════════════════════════════

DM меняет состояние комнаты, рассылается всем (room-schema.json уже есть).
- Сменить активную карту/локацию; туман войны (fog в схеме); режим
  Исследование/Бой (тот индикатор слева — сейчас заглушка, оживёт от сервера);
  фоновая музыка/аудио (audio в схеме).
- Все эти изменения — состояние комнаты, broadcast всем; слать может только is_gm.
События: scene.set, fog.update, mode.set, audio.set (все gating на is_gm).

═══════════════════════════════════════════════════════════════════════
БЛОК 4 — Приватный канал DM ↔ игрок
═══════════════════════════════════════════════════════════════════════

Это «Запросы к DM'у», которые мы скрыли (пункт 7). Парная половина частично есть
на сервере (request.toGm, whisper.open).
- Сторона DM: входящие приватные запросы от игроков (скрытная проверка, личный
  вопрос, отойти от группы) + возможность ответить лично / открыть приватный бросок.
- Скрытная проверка смыкается с блоком 1: DM может прислать ПРИВАТНЫЙ запрос
  проверки, результат которого виден только ему и игроку, не всей партии.
События: переиспользуют request.toGm/whisper.open + check.* в приватном режиме.

═══════════════════════════════════════════════════════════════════════
СВЯЗИ МЕЖДУ БЛОКАМИ (почему порядок важен)
═══════════════════════════════════════════════════════════════════════
- Блок 1 опирается на: карточки персонажей (модификаторы спасбросков/навыков),
  вдохновение в resources. → нужен персонаж из БД (серия лобби) и панель ресурсов.
- Блок 2 опирается на: токены в клетках (пункт 3) + рендер карточки (для мини-листа
  NPC). Готовит трекер инициативы (пункт 4).
- Блок 3 опирается на: room-schema (есть) + режим боя (заглушка). Смыкается с боем.
- Блок 4 опирается на: request.toGm/whisper.open (частично есть) + блок 1
  (приватные проверки).
- СКВОЗНОЕ: ролевое ветвление UI (gm|player) и «пульт мастера» — фундамент под все
  блоки, делается ПЕРВЫМ.

═══════════════════════════════════════════════════════════════════════
ПОЭТАПНЫЙ ПЛАН (реализуем по одному этапу)
═══════════════════════════════════════════════════════════════════════

ЭТАП 0 — Ролевое ветвление + каркас «пульта мастера» (фундамент)
  Клиент ветвит рендер по role: игроку — бар слотов (есть), мастеру — пустой
  «пульт мастера» на месте бара (пока кнопки-заглушки: Проверка / NPC / Сцена /
  Режим). Сервер уже знает is_gm. Без логики — только каркас и ветвление.
  Зачем первым: на этот каркас вешаются все блоки; без него каждый блок будет
  тащить ветвление за собой.
  СТАТУС: сделано (role-ветвление в prototype.html, dm-console заглушки,
  require_gm на сервере).

ЭТАП 1 — Блок 1, базовая проверка (DM просит → игрок кидает → результат)
  Без вдохновения и без свёртывания сначала: DM выбирает игрока+характеристику+DC,
  сервер шлёт запрос, у игрока окно с кнопкой «Бросить», сервер кидает+сравнивает,
  результат к DM и в журнал. Pending-состояние на сервере.

ЭТАП 2 — Блок 1, доработка (свёртывание окна + переброс за вдохновение + партия)
  Окно сворачивается-но-не-закрывается; pending переживает reconnect; переброс
  тратит вдохновение (серверная операция); выбор «вся партия»/несколько; таблица
  результатов у DM.

ЭТАП 3 — Блок 2, NPC-фишки
  DM создаёт NPC (статблок по character-schema, npc:true); фишка на карте —
  портрет; клик — мини-лист. controlledBy на токенах; token.move с проверкой прав.

ЭТАП 4 — Блок 3, управление сценой
  Смена карты, туман, режим Исследование/Бой (оживить индикатор), аудио. Всё
  gating на is_gm, broadcast всем.

ЭТАП 5 — Блок 4, приватный канал
  Входящие запросы к DM (вернуть скрытый UI пункта 7), приватные проверки
  (блок 1 в режиме «виден только DM+игроку»), личные ответы.

Каждый этап даёт играбельный результат и проверяется до следующего. Этапы 1–2 —
самые ценные для игры (проверки — то, что DM делает постоянно), поэтому идут сразу
после фундамента.
